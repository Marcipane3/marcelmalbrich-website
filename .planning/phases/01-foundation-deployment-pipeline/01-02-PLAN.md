<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| GitHub repo → Vercel | Vercel's GitHub App integration gets read access to the repo to trigger builds — standard, scoped, Vercel-managed |
| Internet → marcelmalbrich.com | Public DNS + public deployment — anyone can reach the live site; this phase's content is a static placeholder with no user input, so attack surface is minimal |
| Cloudflare DNS management | If Claude is given Cloudflare API token access, that token must be scoped narrowly (DNS edit on this zone only) and never committed to the repo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-01-03 | Spoofing | DNS record pointing marcelmalbrich.com at Vercel | mitigate | Use the exact record Vercel's `domains add` output specifies (verified ownership via Vercel's own domain verification flow) — do not hand-type an IP from memory; this prevents misconfigured records pointing at the wrong target |
| T-01-04 | Information Disclosure | Cloudflare API token (if used) | mitigate | Token (if provided) is used only in the CLI session, never written to a file in the repo, never committed; if Marcel provides a token, recommend it be scoped to "Zone: DNS Edit" for this single zone only, not account-wide |
| T-01-05 | Tampering | Double-proxy conflict (Cloudflare orange-cloud + Vercel CDN both proxying) | mitigate | Task 3 explicitly sets the Vercel-facing DNS record to "DNS only" (grey cloud) in Cloudflare to avoid SSL/cert and routing conflicts between the two CDNs |
| T-01-06 | Denial of Service | Public placeholder page with no rate limiting | accept | Static page with no backend, no forms, no database — Vercel's platform-level DDoS protections are the baseline; no additional mitigation needed at this phase for a no-interaction placeholder |

</threat_model>
